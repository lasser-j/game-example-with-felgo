Refactoring to C++: A Model/View Architecture with GameController

This tutorial builds on the original Invasion tutorial. If you haven't built the QML-only version yet, start there first. The complete source code for this version is available at GitHub (tag v2.0-cpp-refactor), the original QML-only version remains available at tag v1.0-qml-only.

Motivation

The original tutorial implements all game logic directly in QML/JavaScript: the score, the difficulty curve, the spawn timer, and the movement-vector calculation all live as properties and functions on GameScene. This is a fast way to prototype a game, but it has downsides once a project grows:

  • Game state and UI state are mixed in the same file
  • JavaScript logic cannot be unit-tested without a running QML engine
  • There is no clear ownership boundary between "what the game does" and "how it looks"

In this tutorial, we extract the game logic into a C++ class, GameController, following a Model/View separation: GameController is the model (game state and rules), GameScene.qml and the entity files remain the view (rendering, animation, Felgo-specific APIs).

Architecture Overview

Not everything can move to C++. Felgo's EntityManager, CircleCollider, and MovementAnimation are QML types with no public C++ API for instantiation or collision callbacks. The following table summarises the split:

ResponsibilityWhere it livesWhy
Score, game-running stateC++ (GameController)Pure state, no Felgo dependency
Difficulty curve / spawn intervalC++ (GameController)Pure arithmetic, testable
Movement vector calculationC++ (GameController)Pure arithmetic, testable
Spawn timing and positionC++ (GameController)Timer + arithmetic
Entity creation/removalQML (EntityManager)Felgo-internal, no C++ API
Collision detectionQML (CircleCollider)Felgo-internal, no C++ API
Movement animation playbackQML (MovementAnimation)Felgo-internal, no C++ API

The result is a hybrid architecture: GameController owns all game rules and state, while QML remains responsible for instantiating and animating Felgo entities based on data and signals it receives from GameController.

Implementing GameController

Header

src/GameController.hpp declares the public interface QML will interact with. Properties use Q_PROPERTY with a NOTIFY signal so QML bindings update automatically when the underlying value changes. Methods that QML needs to call directly — such as startGame(), gameOver(), or calculateMovementParameters() — are marked Q_INVOKABLE. Without this macro, a method exists in C++ but is invisible to the QML engine; Q_INVOKABLE exposes it through Qt's meta-object system so it can be called like a regular JavaScript function, e.g. gameController.startGame() or gameController.calculateMovementParameters(x, y, destX, destY).

class GameController : public QObject
{
    Q_OBJECT

    Q_PROPERTY(bool gameRunning     READ isGameRunning  NOTIFY gameRunningChanged)
    Q_PROPERTY(unsigned int  score  READ score          NOTIFY scoreChanged)

public:
    explicit GameController(QObject* parent = nullptr);

    // property readers
    bool isGameRunning() const { return m_gameRunning; }
    unsigned int score() const { return m_score; }

    // invokable game actions

    // resets and (re)starts the game. Call from QML on mouse click.
    Q_INVOKABLE void startGame();
    // stops the game loop. Called internally or from QML on player death.
    Q_INVOKABLE void gameOver();
    // Increments score and recalculates difficulty.
    Q_INVOKABLE void incrementScore();
    // delay for game over to not accidentally dismiss the screen
    Q_INVOKABLE bool isDelayActive() const { return m_delayTimer.isActive(); }

    // set the player position for spawn calculation
    Q_INVOKABLE void updatePlayerPosition(const qreal x, const qreal y, const qreal w, const qreal h);
    // set scene dimensions for spawn calculation
    Q_INVOKABLE void setSceneDimensions(const qreal w, const qreal h);

    // calculates normalised direction vector from origin towards destination.
    Q_INVOKABLE QVariantMap calculateMovementParameters(
        qreal x, qreal y, qreal destX, qreal destY) const;

signals:
    void gameRunningChanged();
    void scoreChanged();

    // emitted by the spawn timer to create enemies in QML's EntityManager.
    void spawnEnemyRequested(const QVariantMap properties);

private:
    // game state
    bool m_gameRunning      = false;    // pause the game until player clicks
    unsigned int m_score    = 0U;       // score dependent on how many enemies are shot

    // constants
    static constexpr int k_initEnemySpawnInterval = 500; // initial spawn interval for enemies in ms
    static constexpr int k_minEnemySpawnInterval  = 200; // // fastest spawn interval for enemies in ms
    static constexpr int k_difficultyStep    = 20;   //  spawn interval for enemies will be increased each difficultyStep score points
    static constexpr int k_spawnAcceleration = 50;   // acceleration for spawning enemies each difficultyStep

    static constexpr int k_delayInterval = 500;  // delay for game over to not accidentally dismiss the screen in ms

    // scene dimensions (used for spawn calculation)
    qreal m_sceneWidth  = 0.0;
    qreal m_sceneHeight = 0.0;

    // player position (used for spawn calculation)
    qreal m_playerX = 0.0;
    qreal m_playerY = 0.0;
    qreal m_playerW = 0.0;
    qreal m_playerH = 0.0;

    // timers
    QTimer m_delayTimer; // delay timer for game over to not accidentally dismiss the screen
    QTimer m_spawnTimer; // spwaner for enemies

    // triggered by spwan timer
    void onSpawnTimerTriggered();

    // private helper functions
    void setGameRunning(const bool v);
    void setScore(const unsigned int v);
    void recalculateDifficulty();
};

Note the two timers replace the two QML Timer elements from the original tutorial: m_spawnTimer corresponds to the enemy spawner, m_delayTimer to the game-over click-guard.

Implementation

The constructor wires up the spawn timer. Both timers' intervals are set once here; m_spawnTimer's interval is later adjusted dynamically by recalculateDifficulty().

GameController::GameController(QObject* parent)
    : QObject(parent) {
    // Delay timer: single-shot, guards against accidental restart clicks.
    m_delayTimer.setSingleShot(true);
    m_delayTimer.setInterval(k_delayInterval);

    // Spawn timer: fires repeatedly while game is running.
    // interval is updated dynamically in recalculateDifficulty().
    m_spawnTimer.setInterval(k_initEnemySpawnInterval);
    connect(&m_spawnTimer, &QTimer::timeout,
            this, &GameController::onSpawnTimerTriggered);
}

startGame() replaces the QML resetGame() logic for resetting score and restarting the spawn timer:

void GameController::startGame() {
    setScore(0U); // reset score
    setGameRunning(true); // reset state

    // reset spawn timer
    m_spawnTimer.setInterval(k_initEnemySpawnInterval);
    m_spawnTimer.start();
}

void GameController::gameOver() {
    setGameRunning(false);
    m_spawnTimer.stop();
    m_delayTimer.start();
}

void GameController::incrementScore() {
    setScore(m_score + 1U);
    recalculateDifficulty();
}

calculateMovementParameters() is a direct C++ port of the JavaScript function from the original tutorial — same inputs, same output map, now type-checked and unit-testable:

QVariantMap GameController::calculateMovementParameters(
    qreal x, qreal y, qreal destX, qreal destY) const {
    // calculate direction from origin towards destination
    qreal dx = destX - x;
    qreal dy = destY - y;
    const qreal len = qSqrt(dx * dx + dy * dy);
    dx /= len;
    dy /= len;

    // create properties for movement animation
    // x/y: start position of bullet - player position moved slightly into direction of mouse click
    // velocityX/Y: direction towards mouse position, used for moving the bullet
    return QVariantMap{
        {"x",         x},
        {"y",         y},
        {"velocityX", dx},
        {"velocityY", dy}
    };
}

The spawn timer's onSpawnTimerTriggered() slot reproduces the random side-selection from the original QML Timer.onTriggered, but uses QRandomGenerator instead of Math.random() and emits a signal instead of calling entityManager directly:

void GameController::onSpawnTimerTriggered() {
    // enemies should appear on a random side, except at the bottom
    const int side = static_cast<int>(
        QRandomGenerator::global()->bounded(3));

    qreal x = 0.0;
    qreal y = 0.0;

    switch (side) {
    case 0: // top
        x = QRandomGenerator::global()->generateDouble() * m_sceneWidth;
        y = -m_playerW;
        break;
    case 1: // left
        x = -m_playerW;
        y = QRandomGenerator::global()->generateDouble() * m_sceneHeight;
        break;
    default: // right
        x = m_sceneWidth + m_playerW;
        y = QRandomGenerator::global()->generateDouble() * m_sceneHeight;
        break;
    }

    // Aim at the centre of the player
    const qreal destX = m_playerX + m_playerW / 2.0;
    const qreal destY = m_playerY + m_playerH / 2.0;

    emit spawnEnemyRequested(calculateMovementParameters(x, y, destX, destY));
}

Finally, recalculateDifficulty() reproduces the difficulty formula from the original QML Timer, applying the new interval directly to m_spawnTimer:

void GameController::recalculateDifficulty() {
    // calculate new spawn frequency (enemies spawn faster depending on score)
    const int difficulty = qFloor(m_score / k_difficultyStep);
    const int newInterval = qMax(k_minEnemySpawnInterval,
                                 k_initEnemySpawnInterval - difficulty * k_spawnAcceleration);

    if(newInterval != m_spawnTimer.interval())
        m_spawnTimer.setInterval(newInterval);
}

Property setters and NOTIFY signals

m_gameRunning and m_score are only ever written through setGameRunning() and setScore(). Both follow the same pattern: update the member, then emit the corresponding NOTIFY signal — but only if the value actually changed.

void GameController::setGameRunning(const bool v) {
    if (m_gameRunning == v) return;
    m_gameRunning = v;
    emit gameRunningChanged();
}

void GameController::setScore(const unsigned int v) {
    if (m_score == v) return;
    m_score = v;
    emit scoreChanged();
}

This is not boilerplate that can be skipped. QML bindings such as

secondaryText: "score: " + gameController.score

and

visible: !gameController.gameRunning

only re-evaluate when the property's NOTIFY signal fires. Without emit gameRunningChanged() / emit scoreChanged(), the values exposed to QML via Q_PROPERTY would still be readable through isGameRunning() and score(), but the bindings above would never update after their initial evaluation — the score text and the pause/game-over overlay would freeze at their first values.

The early-return guard (if (m_score == v) return;) avoids emitting redundant signals when a setter is called with a value that hasn't changed, which matters here because startGame() calls setScore(0U) on every restart, and incrementScore() calls setScore(m_score + 1U) on every hit.

Registering GameController in QML

Unlike EntityBase-derived QML types, GameController is not registered as an instantiable QML type. Instead, a single GameController instance is created in main.cpp and exposed to the whole QML engine as a context property:

// make GameController object available in QML with name "gameController"
GameController controller;
engine.rootContext()->setContextProperty("gameController", &controller);

This makes gameController available as a global identifier in every QML file loaded by this engine — GameScene.qml and all entity files reference it directly, without an import statement and without declaring a GameController {} element anywhere in QML.

Updating GameScene.qml

Scene setup

GameScene.qml no longer declares gameRunning, score, and the various interval properties itself — these now live in the gameController context property set up in main.cpp. On startup, the scene dimensions are passed to GameController once via Component.onCompleted:

Scene {
    id: gameScene

    // init, set scene dimension for spawn calculation
    Component.onCompleted: {
        gameController.setSceneDimensions(
            gameWindowAnchorItem.width,
            gameWindowAnchorItem.height)
    }

    PhysicsWorld { } // no need to set gravity, the collider is not physics-based

    // background image
    Image {
        anchors.fill: gameWindowAnchorItem
        fillMode: Image.PreserveAspectCrop
        source: "../assets/img/background.png"
    }
    // ...
}

Score and overlay bindings

The score display and pause/game-over overlay now bind directly to GameController's NOTIFY-enabled properties:

OverlayText {
    id: scoreText
    anchors.horizontalCenter: gameWindowAnchorItem.horizontalCenter
    y: 10
    secondaryText: "score: " + gameController.score
}

OverlayText {
    id: overlayText
    anchors.centerIn: gameWindowAnchorItem
    visible: !gameController.gameRunning

    primaryText: "pause"
    secondaryText: "click to start"
}

Because gameRunningChanged() and scoreChanged() are emitted from C++, these bindings re-evaluate automatically — no manual property assignment needed.

Reacting to game-over: renaming the overlay text

In the original tutorial, resetGame() directly assigned the "GAME OVER" text. Now that gameOver() is triggered from C++, QML reacts to the resulting gameRunningChanged() signal via Connections:

Connections {
    target: gameController

    // reaction to gameOver() to change overlay-text for next restart.
    function onGameRunningChanged() {
        if (!gameController.gameRunning) {
            // rename overlay text for next game over
            overlayText.primaryText   = "GAME OVER"
            overlayText.secondaryText = "click to restart"
        }
    }

    // create enemies
    function onSpawnEnemyRequested(properties) {
        var enemyID = entityManager.createEntityFromUrlWithProperties(
            Qt.resolvedUrl("entities/Enemy.qml"), properties)
        var entity = entityManager.getEntityById(enemyID)
        entity.active = Qt.binding(function() { return gameController.gameRunning })
    }
}

Spawning enemies: from QML Timer to C++ signal

The original QML Timer.onTriggered block — which calculated the spawn position and called entityManager.createEntityFromUrlWithProperties() directly — is replaced by the onSpawnEnemyRequested handler shown above. GameController performs the calculation and emits spawnEnemyRequested(properties); QML's only remaining job is the actual Felgo entity instantiation.

The Qt.binding() call deserves explanation: createEntityFromUrlWithProperties() only sets properties once, at creation time — it cannot establish a live binding. Qt.binding(function() {...}) creates that binding retroactively on the newly created entity, so its active property continues to track gameController.gameRunning for its entire lifetime. The same pattern is applied to bullets, see below.

Shooting: delegating the vector math

The onPressed handler now delegates the direction calculation to gameController.calculateMovementParameters() instead of a local JavaScript function. The same Qt.binding() pattern used for enemies is applied to bullets:

MouseArea {
    anchors.fill: gameWindowAnchorItem
    onPressed: (mouse)=> {
        if(gameController.gameRunning){
            var movementProperties = gameController.calculateMovementParameters(
                player.x + player.width/2,
                player.y + player.height/2,
                mouse.x,
                mouse.y)

            movementProperties.x += 22*movementProperties.velocityX - 5
            movementProperties.y += 22*movementProperties.velocityY - 5

            var bulletID = entityManager.createEntityFromUrlWithProperties(
                Qt.resolvedUrl("entities/Bullet.qml"), movementProperties)
            var entity = entityManager.getEntityById(bulletID)
            entity.enemyHit.connect(function() { gameController.incrementScore() })
            entity.active = Qt.binding(function() { return gameController.gameRunning })
        }
        else if(!gameController.isDelayActive()) {
            resetGame()
        }
    }
}

Two things changed compared to the original score++: incrementing the score now goes through gameController.incrementScore(), which also triggers the difficulty recalculation in C++. And entity.active is bound the same way as for enemies.

Resetting the game

resetGame() is now minimal — entity cleanup stays in QML (Felgo API), while score reset and restarting the spawn timer move to gameController.startGame():

function resetGame() {
    // delete all enemies and bullets
    entityManager.removeEntitiesByFilter(["enemy","bullet"])

    // (re)start game
    gameController.startGame()
}

Updating the entity files

The active property pattern

Player, Enemy, and Bullet no longer reference gameScene.gameRunning directly. Each entity instead declares its own active property, defaulting to false:

property bool active: false

All running: bindings inside the entity (on GameSpriteSequence and MovementAnimation) reference this local property instead of the scene:

GameSpriteSequence {
    id: spriteSequence
    running: active
    // ...
}

This decouples each entity file from the scene's internal structure: an entity file makes no assumptions about what its parent is called or how it tracks game state. Player is instantiated directly in GameScene.qml with a normal property binding:

Player {
    id: player
    active: gameController.gameRunning
    // ...
}

Enemy and Bullet, being created dynamically via EntityManager, receive their active binding retroactively via Qt.binding() as shown in the previous section.

Player: position reporting

Player.qml additionally reports its initial position to GameController on creation, so the spawn calculation has a valid target from the start:

Player {
    id: player
    active: gameController.gameRunning
    anchors.horizontalCenter: gameWindowAnchorItem.horizontalCenter
    y: gameWindowAnchorItem.height - height - 10

    onHitByEnemy: {
        gameController.gameOver()
    }

    Component.onCompleted: gameController.updatePlayerPosition(
                               player.x, player.y, player.width, player.height)
}

Note: The original tutorial's delay timer is now backed entirely by GameController::m_delayTimer, exposed via isDelayActive(). There is no separate QML delayTimer — gameController.gameOver() starts the C++ delay timer internally, and the MouseArea's else if(!gameController.isDelayActive()) branch (see Shooting: delegating the vector math) consults it directly.

Enemy and Bullet: no further changes

Enemy.qml and Bullet.qml otherwise remain structurally identical to the original tutorial — velocityX/velocityY/speed properties, MovementAnimation, and CircleCollider setup are unchanged, aside from running: active replacing running: gameScene.gameRunning.

What stays in QML, and why

ElementReason it cannot move to C++
EntityManagerFelgo-internal entity lifecycle management; no public C++ API for entity creation/removal
CircleCollider / fixture.onBeginContactFelgo's physics-collision callbacks are QML-only
MovementAnimationFelgo QML animation type; playback is driven by the QML scene graph
Qt.binding() callsRequired because createEntityFromUrlWithProperties() only assigns properties once; live bindings must be established after creation

Result

GameController now owns all game state, the difficulty curve, spawn timing, and movement math — all of it unit-testable with QTest, independent of any running QML engine. QML retains only Felgo-specific entity management, rendering, and the thin active-property glue connecting the two.
